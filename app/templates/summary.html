Lesson Summary

You have reached the end of the FastAPI Teaching Site lesson. Here is a concise review of every concept you explored.

1. What a Backend Does

A backend receives HTTP requests from the browser, reads or writes data in a database, and returns an HTML page or JSON response. FastAPI is the Python framework running this site.

2. GET — Variable Passing via URL

GET sends variables in the URL query string (e.g. /search?q=fastapi or /methods/get-demo?message=hello). These values are visible in the address bar and can be bookmarked. FastAPI reads them from function parameters declared as q: str.

Path parameters (e.g. /users/1) are also GET variables — they appear inside the URL path rather than after ?.

3. POST — Variable Passing via Request Body

POST sends variables in the HTTP request body. The URL does not change. FastAPI reads them using Form(...), which relies on the python-multipart library to parse browser form submissions.

4. HTML Forms

An HTML <form method="post"> collects values in input fields and submits them as POST data. A <form method="get"> appends them to the URL. The action attribute sets which route receives the data.

5. SQLite Database with sqlite3

User records are stored in a SQLite file (teaching.db) using Python's built-in sqlite3 module — no ORM, no separate database server. The users table has id, username, password_hash, and created_at columns. All queries use ? placeholders to prevent SQL injection.

6. Password Hashing

Passwords are never stored as plaintext. hashlib.pbkdf2_hmac("sha256", password, salt, 260_000) produces a hash; a random salt from secrets.token_hex(16) is prepended. The stored string is salt$hash_hex. Verification uses secrets.compare_digest to prevent timing attacks.

7. Login State — Teaching Cookie

After a successful login, FastAPI sets an HTTP-only cookie named teaching_user with the username as the value. Protected routes read request.cookies.get("teaching_user"); if absent, the user is redirected to /login. This is a teaching simplification — production apps use signed session tokens or JWTs, not raw username cookies.

8. Password Change

The change-password route verifies the current password before accepting a new one. If the current password is wrong, the stored hash is unchanged. If correct, a new hash is generated and saved.

9. Logged-in User Creation

The add-user route creates another account, but only when the requester is already logged in. Anonymous requests are redirected to /login. This pattern is common in admin dashboards.

10. Commercial Stack

Real production systems combine a JavaScript frontend (React, Vue, Angular), a backend API (FastAPI, Django, Node/Express), a database (PostgreSQL, MySQL, MongoDB), and a hosting platform (AWS, GCP, Azure) with a reverse proxy (Nginx) and container runtime (Docker/Kubernetes). This demo uses only FastAPI + SQLite to keep the setup learner-friendly.

Full browser-to-backend-to-database flow (register example):
  1. Browser submits POST /register with form body username=alice&password=secret.
  2. FastAPI reads the fields using Form(...).
  3. hash_password("secret") generates salt$pbkdf2_hash.
  4. database.create_user("alice", hashed) runs INSERT INTO users ....
  5. FastAPI returns a 303 Redirect to /login.
  6. Browser follows the redirect and renders the login page.
